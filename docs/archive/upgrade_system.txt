# Client-Server-Commander: System Reloading, Hibernation, and Upgrade Framework

## Abstract

This document proposes a unified framework for *live module reloading*, *system hibernation*, and *incremental upgrades* within the **Client-Server-Commander (CSC)** ecosystem.
The goal is to allow both clients (including AI clients such as Gemini) and servers to safely reload updated code, apply upgrades, or restore a prior system state without interrupting operations or losing data in transit.

---

## 1. Overview

The CSC runtime operates as a long-lived distributed process network built around UDP-based messaging, persistent JSON data storage, and modular service loading.
To maintain uptime and flexibility in production, the system must:

1. Reload changed modules (e.g., `service/*.py`, `server_message_handler.py`) without full restart.
2. Enter a *paused* state during maintenance or upgrade operations.
3. Save complete runtime state to disk (*hibernation*) and restore it after restart.
4. Apply version-controlled upgrades between releases, including structural data migrations.

These features must be **agnostic to execution location** (local or remote), **resilient to partial failure**, and **transparent to connected clients**.

---

## 2. Core Concepts

### 2.1. System States

| State           | Description                                                                                  |
| --------------- | -------------------------------------------------------------------------------------------- |
| **RUNNING**     | Normal operation; processing all commands and file writes.                                   |
| **PAUSED**      | Accepts only keepalives; replies to all other commands with the standardized PAUSED message. |
| **HIBERNATING** | Actively serializing state data to disk; no command processing.                              |
| **UPGRADING**   | Running upgrade scripts and schema migration logic.                                          |
| **RESTORING**   | Loading saved state and resuming normal operation.                                           |

Transitions between these states are controlled by an internal state machine shared by server and client implementations.

---

### 2.2. State Persistence (Hibernation File)

When a node enters **HIBERNATING**, it produces a `state_snapshot.json` (or `.gz` for large states).
The file contains:

```json
{
  "timestamp": 1734830400,
  "system_version": "1.4.2",
  "active_clients": { "<addr>": { "name": "alpha", "last_seen": ... } },
  "services": { "motd_service": { "version": "1.2.1", "state": { ... } } },
  "pending_messages": [ ... ],
  "timers": [ ... ],
  "data_files": { "server_data.json": "sha256:..." },
  "uptime": 48213
}
```

Serialization rules:

* Every subsystem implementing `to_state()` / `from_state()` participates.
* Atomic write via temp file + rename to avoid corruption.
* On startup, if a `state_snapshot.json` exists, the process loads it and deletes the file afterward.

---

### 2.3. Pause / Resume Semantics

**Pause Command**

* Broadcast `SYSTEM:PAUSE` to all clients.
* Server sets state = PAUSED.
* All message handlers reject new commands, replying:

  ```
  [Server] System paused for maintenance — try again later.
  ```
* Clients maintain keepalive heartbeat to avoid timeouts.

**Resume Command**

* Broadcast `SYSTEM:RESUME`.
* Server sets state = RUNNING.
* All client `last_seen` timestamps are incremented by the duration of pause, ensuring sessions remain valid.

This provides clean quiescence before hibernation or upgrade.

## 3. Module Reloading Framework

### 3.1. Motivation

Service modules under `/services/` may change frequently during development or hot-fixes. Reloading them at runtime eliminates full system restarts.

### 3.2. Mechanism

Each module loaded by the `Services` class is tracked by filename and modification timestamp.

```python
self.loaded_modules = {
    "motd_service": {"module": <module>, "mtime": 1734830410}
}
```

**Periodic check (e.g., every 10 seconds):**

1. Compare current `os.path.getmtime(file)` with stored value.
2. If newer, unload using `importlib.invalidate_caches()` and `importlib.reload()`.
3. Replace the entry in `self.loaded_modules`.
4. Log reload event and optionally broadcast `[Server] Service motd_service reloaded.`

All dependent objects must access these modules dynamically to ensure the reloaded code is used without stale references.

---

### 3.3. Safety Guards

* Only reload modules with identical public interfaces (same class/function names).
* If reload fails, revert to previous version from in-memory cache.
* Never reload while PAUSED = False and pending commands exist.

## 4. Upgrade System

### 4.1. Upgrade Package Layout

```
upgrade/
  ├── 1.4.2_to_1.4.3/
  │     ├── upgrade.json
  │     ├── scripts/
  │     │     ├── pre_migrate.py
  │     │     ├── migrate_data.py
  │     │     └── post_verify.py
  │     └── rollback.json
  └── failed_upgrades/
```

`upgrade.json` defines transformation steps and version mappings.

### 4.2. Upgrade Workflow

1. **Detection:**
   Upgrade manager detects new folder under `upgrade/` with a version greater than current.
2. **Pre-Check:**
   Validate required files and compatible base version.
3. **Pause System:**
   Issue `SYSTEM:PAUSE`, wait for quiescence.
4. **Hibernation Snapshot:**
   Save full system state for rollback.
5. **Run Upgrade Scripts:**
   Execute `pre_migrate.py`, `migrate_data.py`, and `post_verify.py`.
6. **Schema Transformation:**
   Modify JSON data schemas as defined in `upgrade.json`.
7. **Swap Core Files:**
   Copy updated source files into place.
8. **Restart / Restore:**
   Restart process, load transformed state.
9. **Cleanup:**
   Move upgrade package to `upgrade/completed/` or to `failed_upgrades/` if an error occurred.

---

### 4.3. Rollback Handling

If startup verification fails post-upgrade:

* Automatically revert to previous source versions (kept as `.bak` in `upgrade/rollback.json`).
* Restore pre-upgrade `state_snapshot.json`.
* Log failure and mark the upgrade as **FAILED**.

## 5. Hibernation and Restoration Internals

### 5.1. Save Procedure

1. Acquire global **state lock**.
2. Block new writes, let in-flight operations complete.
3. Each subsystem returns serializable state:

   * Clients → connection map
   * Services → cache/state
   * File handlers → open file lists
   * Network layer → pending sends
4. Write combined snapshot to disk atomically.
5. Release lock.

### 5.2. Load Procedure

1. Read snapshot file.
2. Recreate subsystem objects.
3. Restore timers and sessions.
4. Delete snapshot file.
5. Transition state = RUNNING.

---

## 6. Interaction Between Hibernation and Upgrade

| Stage              | Hibernation Use                                    |
| ------------------ | -------------------------------------------------- |
| **Before Upgrade** | Captures full live system for rollback.            |
| **During Upgrade** | Inactive — ensures no state drift.                 |
| **After Upgrade**  | Used again to resume identical session continuity. |

This coupling ensures upgrades are atomic and reversible.

---

## 7. Security and Integrity

* **Signatures:** upgrade and snapshot files can be signed with an RSA key (e.g., Easy-RSA CA).
* **Hash Verification:** verify SHA-256 of all core modules before reload or upgrade.
* **Isolation:** the upgrade process runs under limited privileges, ideally in a sandbox or subprocess.

---

## 8. Future Enhancements

1. **Cross-Machine State Migration:**
   Serialize full state and send to another host to clone or fail-over a live node.
2. **Incremental Hibernation:**
   Use diff-based persistence to snapshot only deltas since last save.
3. **Event Journal:**
   Append journal entries for deterministic replay rather than full snapshot restore.
4. **Cluster-Aware Coordination:**
   Extend pause/hibernate/upgrade semantics across distributed nodes.

---

## 9. Operational Summary

| Operation          | Trigger                           | Effect                                        | Client Impact            |
| ------------------ | --------------------------------- | --------------------------------------------- | ------------------------ |
| `system pause`     | Manual / Upgrade Pre-Step         | Stops command processing, maintains keepalive | Receives pause message   |
| `system resume`    | Manual / Post-Upgrade             | Resumes normal operation                      | Resumes heartbeat        |
| `system hibernate` | Manual / Pre-Shutdown             | Serializes runtime state                      | Temporary disconnect     |
| `system restore`   | Auto on startup if snapshot found | Restores full session state                   | Transparent reconnect    |
| `system upgrade`   | Detected new version              | Pauses, hibernates, runs upgrade, restarts    | Short maintenance window |

## 10. Conclusion

This design provides a **robust live-upgrade and state-management framework** for the Client-Server-Commander ecosystem.
It balances flexibility (hot module reloads), reliability (stateful hibernation), and maintainability (structured upgrades).
When implemented, CSC nodes can evolve continuously, apply updates in place, and recover instantly from restarts without losing synchronization or context.
