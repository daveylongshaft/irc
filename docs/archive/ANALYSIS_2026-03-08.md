### INCONSISTENCY 3: Dynamic Service Loading — Implemented but README Omits the Approval Step (Minor)

**What docs say** — `irc/README.md` line 11 and line 59:
> "AI agents can upload new Python code directly to the server's `services/` directory using the
> `<begin file>` protocol. These modules are dynamically loaded and immediately executable."

**What actually exists** — The feature is **fully implemented**, but the README omits the mandatory
approval step. The actual three-step flow is:

1. **Upload** — `<begin file=backup>` / `<end file>` triggers `server_file_handler.py`
   (`FileHandler.start_session` → `complete_session`). The file is written to `staging_uploads/`
   (not directly to `services/`). AST validation runs: must contain exactly one correctly-named
   class.

2. **Approve** — `AI <token> module_manager approve backup` calls
   `module_manager_service.py:approve()`, which re-validates the staged file and moves it to
   `services/` (with versioned backup of any prior file). This step is required; files in
   `staging_uploads/` are not loaded.

3. **Rehash** — `AI <token> module_manager rehash backup` calls `module_manager_service.py:rehash()`
   which uses `importlib.reload()` (or `importlib.import_module()` for new modules) to load the
   service live. At that point it is available via the `AI` dispatcher.

Additional `module_manager` commands: `list`, `staging`, `reject`, `read`, `create` (base64 upload).

**Impact** — Feature works, but the README saying "immediately executable" misleads agents and
developers into thinking no approval is needed. An agent attempting to upload and call a service
without the approval step will get a "service not found" error.

**Where to fix** — `irc/README.md` lines 11 and 59 should describe the staging → approve → rehash
workflow. `docs/services.md` should be updated to match.

### INCONSISTENCY 4: Process Model — Unified Daemon vs. Separate Executables (Moderate)

**What docs say** — `irc/README.md` Quick Start and `docs/setup.md` describe starting each
component separately (e.g., `csc-server`, `csc-client`, `csc-gemini`). Cron or scheduled task
setup is implied for test runner and queue worker.

**What exists** — `irc/packages/csc-service/csc_service/main.py` implements a **single unified
daemon** (`--daemon` flag) that starts the IRC server as a background thread and runs the infra
cycle (test_runner → queue_worker → pm → git_sync) in a single poll loop with configurable
`poll_interval` (default 60s). Separate entry points still exist via console scripts in
`pyproject.toml`, but the intended runtime model is the unified daemon.

**Impact** — Moderate. Ops agents setting up the service per docs will create separate processes
when the correct approach is one daemon process. Systemd unit files in `irc/deploy/` may be out
of date.

**Where to fix** — `irc/README.md` Quick Start section, `docs/setup.md`, `docs/library/SERVICE_SETUP.md`,
systemd unit templates in `irc/deploy/`.
